# AI 写代码时的对话记录，是否应纳入 Git 提交历史？

一个名为 Memento 的开源项目在 Hacker News 引发热议，核心议题是：当 AI 辅助编写代码时，开发者与 AI 的完整对话会话是否应当成为代码提交（commit）的一部分？

![AI 辅助编程与代码历史](https://opengraph.githubassets.com/37c3c32af5ad0ba124496e76012046174436af51f71559557bec6ba39982e0b5/mandel-macaque/memento)

## 事件经过

GitHub 项目 Memento（mandel-macaque/memento）提出了一个工程实践问题：随着 AI 编程助手（如 GitHub Copilot、Cursor、Claude Code 等）越来越深度地参与代码生成，传统的 Git 提交记录只保留了最终代码，却丢失了"为什么这样写"和"怎样得出这段代码"的全过程。

该项目探索将 AI 对话会话作为元数据附加到 Git 提交中，从而保留完整的决策路径：提了什么问题、AI 给出了哪些方案、开发者选择了哪个、中间经历了哪些迭代。

这一话题在 Hacker News 上引发讨论，赞成者认为这对代码审查、知识传承和未来维护有实质性帮助；反对者则担忧隐私问题（对话中可能含有敏感业务逻辑）和仓库膨胀。

## 背景与意义

软件工程界长期倡导"提交信息要解释'为什么'而非'做了什么'"，但 AI 时代的代码生成让这一挑战变得更加复杂。当一段代码是由 AI 在数轮对话中逐步生成的，传统提交信息很难完整捕捉这一过程。

从审计和合规角度看，记录 AI 的决策参与路径也越来越重要——特别是在金融、医疗等受监管行业，代码变更的"可解释性"有明确要求。

另一方面，这也引出了关于代码著作权的深层问题：大量由 AI 生成的代码，其版权归属依然是悬而未决的法律议题。

## 后续关注点

- GitHub、GitLab 等代码托管平台是否会原生支持 AI 会话记录的关联存储？
- 企业在合规审计方面，是否应当强制要求记录 AI 辅助编程的完整过程？
- 这一实践会对代码质量审查和技术债务管理产生怎样的影响？
